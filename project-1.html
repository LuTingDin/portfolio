🛒 購物網站

多頁面購物平台，支援商品瀏覽、購物車管理與結帳流程。

為什麼做這個專案？

作為第一個完整前端專案，目標不只是完成購物網站，而是理解多頁面應用程式如何管理資料與 UI 狀態。 在沒有框架的情況下，練習用原生 JavaScript 建立資料流、操作 DOM，以及設計可維護的檔案結構。

  • DOM 操作：動態生成商品列表、購物車內容與價格資訊
  • 狀態管理：理解資料與畫面同步更新的流程
  • localStorage 應用：讓購物車資料在瀏覽器關閉或重新整理後仍保留
  • 多頁面架構：處理商品頁、購物車頁與結帳頁之間的資料共享
  • 響應式設計：確保在手機、平板、桌面裝置都能呈現

主要功能

  • 商品列表瀏覽與分類展示
  • 商品加入購物車
  • 購物車的商品數量調整與刪除商品
  • 即時計算訂單總金額
  • localStorage 狀態保存
  • 結帳表單與訂單確認頁面

技術決策

資料與 UI 分離

商品資料與購物車資料集中管理， UI只負責根據資料渲染畫面。 當購物車內容變動時， 先更新資料，再重新渲染畫面， 讓資料成為唯一可信來源（ Single Source of Truth）。

localStorage 保存購物車狀態

使用 localStorage儲存購物車內容， 即使重新整理頁面或關閉瀏覽器， 使用者仍能保留先前加入的商品， 提升真實電商網站的使用體驗。

模組化檔案結構

將商品資料與共用工具函式集中於 data.js， 個頁面維持獨立的 JavaScript 檔案， 避免程式碼過度集中造成維護困難。

挑戰與解決方案

1. 動態渲染後事件失效

問題： 初期經常在 DOM 尚未產生完成時就綁定事件， 導致按鈕點擊沒有反應。

解決： 先完成 DOM 渲染， 再統一綁定事件監聽器， 理解 JavaScript 執行順序與瀏覽器渲染流程。

2. 購物車資料同步

問題： 商品數量變更後， 購物車列表、價格總計與 localStorage 容易出現不同步的狀況。

解決： 建立固定流程： 先更新資料 → 寫入 loacalStorage → 重新渲染畫面， 確保所有 UI 都來自同一份資料來源。

未來改進方向

  • 加入商品搜尋與篩選功能
  • 增加商品詳細頁面與圖片輪播
  • 加入使用者登入與訂單紀錄功能
  • 串接後端 API，取代目前的靜態商品資料
